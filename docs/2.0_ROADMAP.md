# RasEyes — Development Roadmap (v2.0)

> **목표:** 1.0 PoC 완성 이후 유지보수·분석 인프라 구축 — 무선 연결, 로그 수집·분석, 경고 이벤트 클립 저장  
> **전제:** 1.0 KPI(E2E < 500ms · 추론 < 60ms · 15+ FPS)를 해치지 않는 범위에서 구현  
> **이력:** `wantToMake.md`의 아이디어 + 1.0 로드맵 미완 항목(6-2 CSV 분석)을 이관하여 구성 (2026-07-18)

---

## Phase 1 · 무선 연결 (WiFi 동글) 🔲
> 목표: 유선 랜 없이 무선으로 `ssh raseyes` 접속 가능하게 만들기

| # | 작업 | 비고 | 상태 |
|---|------|------|------|
| 1-1 | 와이파이 동글 구매 | 리눅스 커널 기본 드라이버 포함 칩셋 필수 (예: RTL8821CU, MT7601U 계열). 초저가 동글은 ARM 보드에서 드라이버 직접 빌드가 필요한 칩이 걸릴 수 있음 | 🔲 |
| 1-2 | 동글 인식 확인 | Orange Pi 5에 꽂고 `lsusb`, `ip link`로 인터페이스 확인 | 🔲 |
| 1-3 | NetworkManager로 집 와이파이 자동 연결 등록 | `nmcli` 사용. 등록 대상은 집 와이파이만 (실외 로그는 귀가 후 동기화) | 🔲 |
| 1-4 | 무선 `ssh raseyes` 동작 확인 | 유선 분리 상태에서 접속·재부팅 후 자동 재접속 확인. `~/.ssh/config`의 IP가 바뀌면 갱신 | 🔲 |

**Phase 1 완료 기준:** 유선 랜 없이 부팅 → 와이파이 자동 접속 → PC에서 `ssh raseyes` 성공.

---

## Phase 2 · 로그 동기화 + CSV 분석 🔲
> 목표: Pi에 쌓인 운영 CSV 로그를 PC로 수집하고, 1.0에서 이관된 6-2 분석 수행  
> **범위 제한:** PC에서 수동 rsync pull만 정식 구현. Pi 쪽 systemd timer 자동 push는 수동 방식이 실제로 불편해진 뒤에 재검토 (범위 제외)

| # | 작업 | 비고 | 상태 |
|---|------|------|------|
| 2-1 | 로그 pull 스크립트 작성 | `scripts/pull_logs.sh` — `rsync raseyes:~/RasEyes/logs/*.csv ./logs_archive/` 한 줄 수준. 전용 데몬 금지 | 🔲 |
| 2-2 | CSV 로그 분석 (1.0의 6-2 이관) | FPS 방어율 · 평균 CPU 온도 · 알람 빈도 분석. 착용 테스트 로그 기반 | 🔲 |
| 2-3 | 분석 결과 문서화 및 UX 개선점 도출 | 오탐지 KPI(< 1회/분) 실측치 확인 포함 | 🔲 |

**Phase 2 완료 기준:** 착용 테스트 후 명령 한 번으로 로그가 PC에 수집되고, CSV 분석 결과가 문서로 정리됨.

---

## Phase 3 · 경고 이벤트 클립 저장 🔲
> 목표: 오탐지 분석·유지보수를 위해 경고 발생 시점 전후 프레임을 저장 (상시 녹화 금지 — 저장소 수명·성능 KPI·프라이버시 고려)

### 설계 확정 사항 (2026-07-18)

| 항목 | 결정 |
|------|------|
| 저장 트리거 | **HIGH 경보 + 시스템 이상 이벤트**(카메라 가림 감지 등). MID는 저장량 문제로 제외, 필요해지면 확장 |
| 저장 형식 | **저빈도 JPEG 시퀀스** — mp4 인코딩 없음 (인코딩 CPU 부하가 추론과 겹치는 것 방지) |
| 링 버퍼 | 추론용 프레임 재활용, 초당 5프레임만 JPEG 압축하여 메모리 보관 |
| 클립 범위 | 이벤트 전 5초 + 후 3초 (이벤트당 약 2~3MB 예상) |
| 메타데이터 | 프레임별 타임스탬프·탐지 결과(레이블, bbox, confidence)를 클립 폴더에 동봉 |
| 보존 한도 | 최대 30개 이벤트 폴더, 초과 시 오래된 것부터 삭제 |
| 상수 관리 | 버퍼 FPS · 전/후 초 · 보존 한도 모두 `config.py` 상수화 |

### 구현 작업

| # | 작업 | 비고 | 상태 |
|---|------|------|------|
| 3-1 | `config.py`에 클립 관련 상수 추가 | 버퍼 FPS(5) · 전 5초/후 3초 · JPEG 품질 · 보존 한도(30) | 🔲 |
| 3-2 | 프레임 링 버퍼 구현 | 메인 루프의 비전 큐 프레임을 5FPS로 샘플링해 JPEG 압축 후 `deque` 보관. 15FPS KPI에 영향 없는지 확인 | 🔲 |
| 3-3 | 이벤트 트리거 → 클립 덤프 | HIGH 경보·가림 감지 시 버퍼 + 후속 3초 프레임을 이벤트 폴더로 저장. 저장은 백그라운드 스레드에서 수행 | 🔲 |
| 3-4 | 보존 한도 rotation 구현 | 이벤트 폴더 30개 초과 시 오래된 폴더 삭제 | 🔲 |
| 3-5 | 테스트 작성 | 링 버퍼 샘플링 주기, 트리거 시 덤프 내용(전/후 범위), rotation 동작 | 🔲 |
| 3-6 | 클립 동기화 | Phase 2의 `pull_logs.sh`에 클립 폴더 포함 | 🔲 |

**Phase 3 완료 기준:** 실제 착용 테스트에서 HIGH 경보 발생 시 전후 클립이 저장되고, PC에서 프레임+메타데이터로 오탐/정탐을 판별할 수 있음. KPI(15FPS, E2E < 500ms) 유지 확인.

## 범위에서 제외한 것 (의도적 결정)

| 항목 | 제외 이유 |
|------|-----------|
| 블루투스 동글 로그 전송 | 느리고 구현 부담이 큼 — 와이파이 동글로 대체 |
| Pi → PC 자동 push (systemd timer) | PC 꺼짐 실패 처리·PC 측 ssh 서버 설정 등 부수 비용. 수동 pull이 불편해지면 재검토 |
| 폰 핫스팟 등록 | 실내 개발 편의가 우선. 실외 현장 확인이 필요해지면 추가 |
| 상시 녹화 / mp4 인코딩 | 저장소 수명·CPU 부하·프라이버시 문제 |
| MID 경보 클립 저장 | 발생 빈도가 높아 저장량 폭증 — HIGH 분석 먼저 |
